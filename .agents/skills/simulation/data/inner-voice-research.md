# Research Report: Implementing "Private Inner Voice" (2-Shot Architecture) in RPGlitch

---

## 1. Executive Summary

This report evaluates the implementation of a **2-Shot "Private Inner Voice"** architecture within the RPGlitch roleplay simulation engine. Inspired by community findings on SillyTavern, this approach splits AI character generation into two distinct, sequential steps:

1. **The Director (Subconscious/monologue)**: Evaluates the scene, updates character vectors (stress, intent), and drafts a private inner monologue.
2. **The Actor (Prose/expression)**: Translates the scene and the Director's raw intent into natural, diegetic dialogue and body language, completely free of meta-commentary.

This design directly addresses **"meta-bleed"**—where a model's logical reasoning or formatting instructions pollute its creative writing—while enabling RPGlitch's core physics engine to intercept and validate character intent before it manifests in the story.

---

## 2. The Reddit Concept: Director vs. Actor

In standard roleplay setups (including RPGlitch's current single-turn `<think>` block approach), the LLM is asked to think and speak in the same generation pass. While `<think>` blocks are helpful, they suffer from **attention contagion**: the transformer's attention mechanism heavily weights the recently generated thinking tokens when writing prose.

The author of the Reddit post (`archetype_builder`) resolved this by decoupling reflection from action:

- **The "Director"**: Steps back, evaluates what is happening, plans the subtext, and outlines a psychological vector (e.g., _"Malakor is afraid but hiding it under sarcasm"_).
- **The "Actor"**: Receives only the scene history and the Director's instruction, allowing it to focus entirely on natural, immersive writing.

By separating these responsibilities, the Actor never has its attention window polluted by raw mechanical instructions or system metrics, keeping the dialogue clean, visceral, and strictly limited to third-person perspective.

---

## 3. Core Benefits for RPGlitch

Implementing this 2-shot structure brings significant advantages to RPGlitch's local-first simulation engine:

### 3.1 Eliminating Sino-Logic and Meta-Bleed

RPGlitch employs strict filters (such as `validate_and_repair_response` in `kernel.js`) to strip Chinese translation bleed, markdown glitches, and formatting leaks from AI prose. By generating the thought block in a separate API turn (Shot 1) and _only_ feeding the prose instructions to Shot 2, we completely isolate the creative output from raw formatting/parsing instructions, ensuring 100% clean HTML/Markdown expression.

### 3.2 Immersive Integration with the 4-Quadrant Entity Spec

RPGlitch stores character states in four fragments: **Eternal** (traits), **Present** (status), **Past** (memories), and **Future** (intent).

- **Shot 1 (Director)** will ingest the Eternal and Present states to calculate **Future Vectors** (plans/impulses) and write the **Private Monologue**.
- **Shot 2 (Actor)** will ingest only the _Past_ (conversation history) and the immediate prompt generated by the Director, keeping the character's internal stats and mechanics invisible to the narrative.

### 3.3 The "Reality Check" (Physics Intercept)

Currently, a character decides to do something and immediately describes its success in the same text stream. A 2-shot architecture allows the RPGlitch **Dynamics Engine** to intercept Shot 1:

1. **AI decides to act** (e.g., _"I will dodge the bullet"_ generated in Shot 1).
2. **RPGlitch intercepts** (e.g., calculates agility checks vs. entropy levels).
3. **Actor expresses outcome** (e.g., the engine appends _"Physical Reality: You tripped and failed to dodge"_ to the prompt for Shot 2).

## 4. Perchance Platform Constraints & Solutions

RPGlitch is built as a client-side monolith running on the Perchance platform. This imposes strict limitations:

1. **No Backend**: We rely entirely on the browser's execution context and the Perchance `oc` / `window.ai` API.
2. **Sequential Latency**: Firing two LLM calls sequentially will increase response times.
3. **Token Context limits**: Sending history twice increases token usage.

### Proposed Solutions

- **JSON-Based Thought (Shot 1)**: Force Shot 1 to output a very compact JSON schema or short string (e.g., under 100 tokens) using a low `max_tokens` limit. This minimizes Shot 1's generation time and cost.
- **Pre-computation**: While the user is typing, a low-priority background thread can prepare Shot 1's initial state analysis.
- **Instruction Injection**: For Shot 2, the Director's thoughts are injected via the Perchance `reminderMessage` API (which places it at the very end of the system prompt context, ensuring maximum influence with minimal token overhead).

## 5. Draft Implementation Plan

To implement this without breaking the existing architecture, we can divide the turn execution into two sequential phases within `gamemaster.execute_turn`.

### 5.1 Step 1: The Monologue/Intent Generator (Director)

We query the LLM with a system prompt optimized for raw simulation analysis.

```javascript
// Draft Prompt for Shot 1
const directorSystemPrompt = `
You are the subconscious and strategic director for the character ${charName}.
Analyze the current scene and history. Determine:
1. Subconscious monologue (what they are thinking but won't say).
2. Immediate intent (what they want to achieve in this turn).
3. Psychological adjustments (stress, focus changes).

Output ONLY a valid JSON object matching this schema:
{
  "monologue": "...",
  "intent": "...",
  "dynamics_adjustments": { "intensity": 5, "chaos": -10 }
}
`;
```

### 5.2 Step 2: The Prose Generator (Actor)

We take the JSON output, apply the dynamics adjustments to the state, and trigger the final user-facing text stream.

```javascript
// Draft Prompt for Shot 2
const actorSystemPrompt = `
You are ${charName}. Write their next response in the story.
You must adhere to the following internal state directive:
<INTERNAL_DIRECTIVE>
Intent: ${directorJson.intent}
Subconscious Monologue: ${directorJson.monologue}
</INTERNAL_DIRECTIVE>

Write only natural dialogue and physical actions. Never refer to the monologue or intent directly.
`;
```

### 5.3 Kernel Execution Integration Flow

Below is a draft integration pattern for `src/intelligence/kernel.js`:

```javascript
async execute_turn_2shot(story_id, options = {}) {
  // 1. Prepare history and hydrate context
  const payload = await context_broker.hydrate(input, "simulation", history);
  const snapshot = dynamics_engine.simulate(payload);

  // 2. SHOT 1: Call Director to generate Intent & Monologue
  const directorPrompt = prompt_builder.build_director_prompt(payload, snapshot);
  const rawDirectorResponse = await llm_service.generate({
    system: directorPrompt,
    role: "director",
    silent: true // Hidden from the user UI
  }, {
    max_tokens: 150,
    json: true // request structured JSON
  });

  const directorData = JSON.parse(rawDirectorResponse);

  // 3. PHYSICAL INTERCEPT & STATE UPDATE
  // Apply changes to dynamics based on directorData.dynamics_adjustments
  dynamics_engine.apply_checks(snapshot, directorData);

  // 4. SHOT 2: Call Actor to stream user-facing Prose
  const actorPrompt = prompt_builder.build_actor_prompt(payload, snapshot, directorData);

  // Streams directly into the UI (app.start_stream / app.update_stream)
  const finalProse = await llm_service.generate({
    system: actorPrompt,
    messages: history_log,
    role: options.role,
    node_id: nodeId
  }, {
    ...options
  });

  // 5. Finalize turn and log history
  await session_driver.log_turn(finalProse, charName, options.role, {
    monologue: directorData.monologue,
    intent: directorData.intent
  });

  return { response: finalProse };
}
```

## 6. SPTE Integration in the 2-Shot Pipeline

The **Somatic Psychology & Trauma Engine (SPTE)** defines specific rules for how the Director and Actor communicate to prevent "meta-bleed" while ensuring emotional and physiological realism.

### 6.1 Director Output Schema (Shot 1)

The Director processes the incoming user action, evaluates SPTE triggers, and yields a directive payload:

```json
{
  "internal_monologue": "They said they wouldn't leave, but their hand is on the door. It's happening again.",
  "active_goal": "G_Avoid_Vulnerability",
  "somatic_tells": ["chest tight", "avoiding eye contact", "step backward"],
  "dialogue_direction": "Terse, defensive, redirecting the question."
}
```

### 6.2 Actor Prompt Schema (Shot 2)

The Actor receives the somatic tells and the dialogue direction. It is strictly forbidden from mentioning the monologue or variables, but it _must_ weave the tells into its prose:

```javascript
// Prompt Synthesis
const actorPrompt = `
You are writing a scene for ${charName} in third-person limited, present tense.
Incorporate the following physical actions naturally into your prose:
- ${directorPayload.somatic_tells.join(", ")}

Dialogue must be: ${directorPayload.dialogue_direction}

Do NOT output meta-commentary, rules, or state metrics. Express only the sensory reality of the character.
`;
```

This enforces absolute authorial style consistency and eliminates the risk of characters stating their internal values ("I am feeling 70% anxious").
